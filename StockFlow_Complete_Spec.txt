SECTION 1 — PRODUCT REQUIREMENTS DOCUMENT (PRD)
================================================================================

1.1 PRODUCT OVERVIEW
--------------------
StockFlow is a multi-tenant SaaS inventory management application for small
sellers and internal teams. Users can track products, monitor stock levels, and
see low-stock alerts through a browser-based interface.

  Product Name  : StockFlow MVP
  Version       : v0.1 (6-Hour Build)
  Target User   : Small sellers, internal teams, warehouse operators
  Deployment    : Single-region, browser-based SaaS
  Build Timeline: ~6 hours for MVP demo


1.2 OBJECTIVES
--------------
- Allow users to sign up, create an organization, and log in securely
- Provide full CRUD operations for product and inventory management
- Display a real-time dashboard with summary metrics and low-stock alerts
- Scope all data by organization to ensure multi-tenant isolation
- Deliver a demo-ready app without external integrations


1.3 IN-SCOPE FEATURES
---------------------

Authentication & Tenant Basics:
  FR-1  : Email/password signup with organization name
  FR-2  : Secure login with JWT or session-based auth
  FR-3  : Organization record created and linked on signup
  FR-4  : All data scoped by organization ID

Product & Inventory Management:
  FR-5  : Product model — ID, Org ID, Name, SKU, Description, Quantity,
          Cost Price, Selling Price, Low Stock Threshold, timestamps
  FR-6  : Create product via form
  FR-7  : List products in a searchable table
  FR-8  : Edit any product field including quantity
  FR-9  : Delete product with confirmation dialog
  FR-10 : Stock adjustment control (inline +/- N units)

Dashboard:
  FR-11 : Total product count card
  FR-12 : Total units on hand card
  FR-13 : Low-stock items table (Quantity <= Threshold)

Settings:
  FR-14 : Global default low-stock threshold setting


1.4 OUT-OF-SCOPE (PHASE 1)
--------------------------
  Feature                          Reason
  -------                          ------
  Multi-warehouse                  Phase 2 — complex logistics
  Product variants (size/color)    Phase 2 — data model complexity
  Shopify / WooCommerce            Phase 3 — OAuth + webhooks needed
  Order management                 Phase 2 — separate domain
  Email notifications              Phase 2 — requires email service
  CSV import/export                Phase 2 — file handling complexity
  RBAC / multi-user orgs           Phase 2 — one user per org in MVP
  Billing / subscriptions          Phase 3 — payment gateway needed
  Audit logs                       Phase 2 — event sourcing overhead


1.5 SUCCESS CRITERIA
--------------------
  1. A new user can sign up, log in, and see a populated dashboard
  2. At least one product can be created with SKU, quantity, and pricing
  3. Product appears correctly in list view and dashboard summary
  4. Low-stock threshold logic triggers correctly and shows alert table
  5. No cross-tenant data leaks between different organizations
  6. Entire flow is demo-able in a browser without manual DB edits

SECTION 2 — SYSTEM OVERVIEW
================================================================================

2.1 APPLICATION CONTEXT
-----------------------
StockFlow is a three-tier web application:
  - React/Next.js SPA (frontend)
  - Node.js/Express REST API (backend)
  - PostgreSQL relational database

2.2 COMPONENT INTERACTION FLOW
-------------------------------

  +----------------------------------------------------------+
  |                  BROWSER (React SPA)                     |
  |  Login/Signup -- Dashboard -- Products -- Settings       |
  +-------------------------+--------------------------------+
                            | HTTPS / REST API
                            v
  +----------------------------------------------------------+
  |            NODE.JS / EXPRESS API SERVER                  |
  |  Auth Middleware -- Routes -- Controllers -- Prisma      |
  +-------------------------+--------------------------------+
                            | SQL (via Prisma ORM)
                            v
  +----------------------------------------------------------+
  |                  POSTGRESQL DATABASE                     |
  |  users -- organizations -- products -- settings          |
  +----------------------------------------------------------+


2.3 KEY DESIGN DECISIONS
------------------------
  Decision          Choice              Rationale
  --------          ------              ---------
  Auth method       JWT (stateless)     No server-side session store needed
  ORM               Prisma              Type-safe, fast schema setup
  Database          PostgreSQL          ACID-compliant, schema enforcement
  Multi-tenancy     Org ID on every row Simple shared-DB, shared-schema
  Frontend          Next.js + Tailwind  Fast dev, built-in routing

SECTION 4 — DATABASE SCHEMA
================================================================================

4.1 ENTITY RELATIONSHIPS
-------------------------
  organizations  --<  users         (one org, one user in MVP)
  organizations  --<  products      (one org, many products)
  organizations  --<  org_settings  (one org, one settings record)


4.2 TABLE: organizations
------------------------
  Column      Type          Constraints          Description
  ------      ----          -----------          -----------
  id          UUID          PK, default gen      Primary key (UUID v4)
  name        VARCHAR(255)  NOT NULL             Organization display name
  slug        VARCHAR(100)  UNIQUE, NOT NULL     URL-safe org identifier
  created_at  TIMESTAMPTZ   DEFAULT NOW()        Record creation timestamp
  updated_at  TIMESTAMPTZ   DEFAULT NOW()        Auto-updated on change


4.3 TABLE: users
----------------
  Column          Type          Constraints     Description
  ------          ----          -----------     -----------
  id              UUID          PK              Primary key
  organization_id UUID          FK → orgs.id    Owning organization
  email           VARCHAR(255)  UNIQUE NOT NULL Login email address
  password_hash   VARCHAR(255)  NOT NULL        bcrypt hashed password
  created_at      TIMESTAMPTZ   DEFAULT NOW()   Registration timestamp
  updated_at      TIMESTAMPTZ   DEFAULT NOW()   Last update timestamp


4.4 TABLE: products
-------------------
  Column              Type           Constraints     Description
  ------              ----           -----------     -----------
  id                  UUID           PK              Primary key
  organization_id     UUID           FK, NOT NULL    Scoping org reference
  name                VARCHAR(255)   NOT NULL        Product display name
  sku                 VARCHAR(100)   NOT NULL        Unique per org (see index)
  description         TEXT           NULLABLE        Optional description
  quantity_on_hand    INTEGER        NOT NULL, >=0   Current stock count
  cost_price          DECIMAL(10,2)  NULLABLE        Purchase/cost price
  selling_price       DECIMAL(10,2)  NULLABLE        Customer-facing price
  low_stock_threshold INTEGER        NULLABLE        Per-product override
  created_at          TIMESTAMPTZ    DEFAULT NOW()   Creation timestamp
  updated_at          TIMESTAMPTZ    DEFAULT NOW()   Last update timestamp

  Unique index: (organization_id, sku)


4.5 TABLE: org_settings
-----------------------
  Column                       Type         Constraints    Description
  ------                       ----         -----------    -----------
  id                           UUID         PK             Primary key
  organization_id              UUID         FK, UNIQUE     One row per org
  default_low_stock_threshold  INTEGER      DEFAULT 5      Org-wide fallback
  updated_at                   TIMESTAMPTZ  DEFAULT NOW()  Last settings update


4.6 PRISMA SCHEMA
-----------------

  model Organization {
    id         String       @id @default(uuid())
    name       String
    slug       String       @unique
    createdAt  DateTime     @default(now()) @map("created_at")
    updatedAt  DateTime     @updatedAt @map("updated_at")
    users      User[]
    products   Product[]
    settings   OrgSettings?
  }

  model User {
    id             String       @id @default(uuid())
    organizationId String       @map("organization_id")
    email          String       @unique
    passwordHash   String       @map("password_hash")
    createdAt      DateTime     @default(now())
    updatedAt      DateTime     @updatedAt
    organization   Organization @relation(fields: [organizationId], references: [id])
  }

  model Product {
    id                String       @id @default(uuid())
    organizationId    String       @map("organization_id")
    name              String
    sku               String
    description       String?
    quantityOnHand    Int          @default(0) @map("quantity_on_hand")
    costPrice         Decimal?     @map("cost_price")
    sellingPrice      Decimal?     @map("selling_price")
    lowStockThreshold Int?         @map("low_stock_threshold")
    createdAt         DateTime     @default(now())
    updatedAt         DateTime     @updatedAt
    organization      Organization @relation(fields: [organizationId], references: [id])
    @@unique([organizationId, sku])
  }

  model OrgSettings {
    id                       String       @id @default(uuid())
    organizationId           String       @unique
    defaultLowStockThreshold Int          @default(5)
    updatedAt                DateTime     @updatedAt
    organization             Organization @relation(fields: [organizationId], references: [id])
  }

SECTION 5 — API SPECIFICATION
================================================================================

5.1 BASE URL & CONVENTIONS
--------------------------
  Base URL     : /api/v1
  Auth Header  : Authorization: Bearer <jwt_token>
  Content-Type : application/json
  Error format : { error: string, details?: any }


5.2 AUTH ENDPOINTS
------------------
  Method  Endpoint       Request Body                    Response
  ------  --------       ------------                    --------
  POST    /auth/signup   { email, password, orgName }    201: { token, user, organization }
  POST    /auth/login    { email, password }             200: { token, user } | 401
  GET     /auth/me       — (JWT required)                200: { user, organization }


5.3 PRODUCT ENDPOINTS (JWT required)
-------------------------------------
  Method  Endpoint                      Request                        Response
  ------  --------                      -------                        --------
  GET     /products                     ?search=&page=&limit=          200: { products[], total, page }
  GET     /products/:id                 —                              200: Product | 404
  POST    /products                     ProductCreateDTO               201: Product | 409: SKU exists
  PATCH   /products/:id                 ProductUpdateDTO (partial)     200: Updated Product
  DELETE  /products/:id                 —                              204: No Content | 404
  POST    /products/:id/adjust-stock    { adjustment, note? }          200: { product, newQuantity }


5.4 DASHBOARD ENDPOINT (JWT required)
--------------------------------------
  Method  Endpoint             Request  Response
  ------  --------             -------  --------
  GET     /dashboard/summary   —        200: { totalProducts, totalUnits, lowStockItems[] }


5.5 SETTINGS ENDPOINTS (JWT required)
--------------------------------------
  Method  Endpoint    Request                               Response
  ------  --------    -------                               --------
  GET     /settings   —                                     200: OrgSettings
  PUT     /settings   { defaultLowStockThreshold: number }  200: Updated OrgSettings


5.6 REQUEST / RESPONSE DTOs
----------------------------

  ProductCreateDTO:
    name              : string   (required)
    sku               : string   (required)
    description       : string   (optional)
    quantityOnHand    : number   (required, >= 0)
    costPrice         : number   (optional)
    sellingPrice      : number   (optional)
    lowStockThreshold : number   (optional)

  ProductUpdateDTO (all fields optional):
    name, sku, description, quantityOnHand,
    costPrice, sellingPrice, lowStockThreshold

  StockAdjustDTO:
    adjustment : number   (positive or negative integer)
    note       : string   (optional)


5.7 ERROR CODES
---------------
  HTTP  Error Key         When
  ----  ---------         ----
  400   VALIDATION_ERROR  Missing required fields or invalid format
  401   UNAUTHORIZED      Missing or invalid JWT
  403   FORBIDDEN         Accessing another org's resource
  404   NOT_FOUND         Resource not found or not in this org
  409   CONFLICT          SKU already exists in this organization
  500   INTERNAL_ERROR    Unhandled server error

SECTION 6 — SECURITY REQUIREMENTS
================================================================================

6.1 AUTHENTICATION SECURITY
----------------------------
  Requirement      Implementation
  -----------      --------------
  Password hashing bcrypt with salt rounds >= 12; never store plaintext
  JWT secrets      Min 256-bit random secret; stored in env variables
  JWT expiry       Access tokens expire in 24 hours
  HTTPS only       All production traffic over TLS
  Input validation Zod validation on all API request bodies and params


6.2 AUTHORIZATION & MULTI-TENANCY
-----------------------------------
  - Every API handler MUST extract organizationId from JWT, never from body
  - All DB queries MUST include WHERE organization_id = :orgId
  - Product IDs verified against requesting org — prevents ID enumeration
  - Middleware enforces org scoping before reaching controller logic


6.3 API SECURITY
----------------
  Control       Details
  -------       -------
  CORS          Restrict to known frontend origin (env-configured)
  Rate limiting Basic rate limit on /auth routes (100 req / 15 min)
  SQL injection Prisma ORM parameterizes all queries by default
  XSS           React DOM escapes output; no dangerouslySetInnerHTML
  Env secrets   DATABASE_URL, JWT_SECRET in .env; never commit to VCS

10.5 QUICK-START COMMANDS
--------------------------

  # 1. Install dependencies
  pnpm install

  # 2. Start PostgreSQL (Docker)
  docker run -e POSTGRES_PASSWORD=secret -p 5432:5432 postgres:16

  # 3. Run DB migrations
  cd apps/api && npx prisma migrate dev --name init

  # 4. Start API server (port 4000)
  cd apps/api && pnpm dev

  # 5. Start frontend (port 3000)
  cd apps/web && pnpm dev

  # 6. Open browser
  open http://localhost:3000
